## Workspace Package Definitions

### audiflow_app (Main App)
- **Type**: Flutter application
- **Dependencies**: All other packages
- **Contains**: App entry points, routing, app-level configuration, presentation layer
- **Path**: `packages/audiflow_app/`

### audiflow_core
- **Type**: Dart package
- **Dependencies**: None (or minimal external deps only)
- **Contains**: Constants, extensions, utilities, error types, logger
- **Path**: `packages/audiflow_core/`

### audiflow_domain
- **Type**: Dart package
- **Dependencies**: `audiflow_core`, `drift`, `dio`, etc.
- **Contains**: Business logic, services, repositories (interfaces + implementations), data sources, Drift models (used as both domain entities and DTOs)
- **Path**: `packages/audiflow_domain/`
- **Note**: This package merges traditional data and domain layers for performance. Drift models serve dual purpose as domain entities and database models.

### audiflow_podcast
- **Type**: Dart package
- **Dependencies**: `audiflow_core`, `xml`, `intl`, `logger`
- **Contains**: Podcast RSS parsing utilities with streaming support
- **Path**: `packages/audiflow_podcast/`
- **Features**:
  - Streaming RSS parser for memory-efficient parsing
  - Comprehensive RSS 2.0 and iTunes namespace support
  - Intelligent caching with HTTP optimization
  - Builder interface for zero-copy entity construction
  - Error resilience with graceful handling of malformed feeds

### audiflow_ui
- **Type**: Flutter package
- **Dependencies**: `audiflow_core`, `audiflow_domain`
- **Contains**: Reusable widgets, themes, styles
- **Path**: `packages/audiflow_ui/`

## Layer Rules (STRICT)

```
App в†’ UI в†’ Domain в†’ Podcast
                       в†“
                     Core (used by all)
```

**Dependency flow:**
- `audiflow_app` depends on: `audiflow_ui`, `audiflow_domain`, `audiflow_core`
- `audiflow_ui` depends on: `audiflow_domain`, `audiflow_core`
- `audiflow_domain` depends on: `audiflow_podcast`, `audiflow_core`
- `audiflow_podcast` depends on: `audiflow_core`
- `audiflow_core` has NO dependencies on other packages

**What goes where:**
- **App**: Main entry points, flavor configs, root providers, routing setup, presentation layer (screens, controllers)
- **UI**: Reusable widgets, themes, styles (used across multiple features)
- **Domain**: Business logic, services, repositories (interfaces + implementations), data sources (Drift, HTTP), models (@collection for Drift, @freezed for non-persisted models)
- **Podcast**: RSS parsing, podcast feed models, caching, network fetching for podcast data
- **Core**: Constants, extensions, utilities, error types, validators, formatters

**Performance-focused design:**
- Drift models serve as both domain entities and database models (no DTO mapping)
- Repository interfaces and implementations live together in `audiflow_domain`
- Direct use of Drift models throughout the app (zero conversion overhead)

### 2. Riverpod Controllers (in audiflow_app or audiflow_ui)
```dart
@riverpod
class PodcastListController extends _$PodcastListController {
  @override
  Future<List<Podcast>> build() async {
    return ref.watch(podcastRepositoryProvider).getSubscriptions();
  }
}
```

### 3. Drift Models (Dual-purpose: Domain Entity + Database Model)
```dart
// In audiflow_domain/src/features/feed/models/podcast.dart
import 'package:drift/drift.dart';

// Table definition
class Podcasts extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get feedUrl => text().customConstraint('UNIQUE')();
  TextColumn get title => text()();
  TextColumn get description => text().nullable()();
  TextColumn get imageUrl => text().nullable()();
  DateTimeColumn get lastFetchedAt => dateTime()();
}

// Note: Data class (Podcast) is auto-generated by Drift
// Use throughout the app - no separate DTOs needed
```

## Decision Tree: Where to Add New Code

| What are you adding? | Package | Location | Example |
|---------------------|---------|----------|---------|
| **New feature presentation** | `audiflow_app` | `lib/features/{feature}/presentation/` | `audiflow_app/lib/features/playlist/presentation/` |
| **Shared widget** (2+ features) | `audiflow_ui` | `lib/src/widgets/` | `audiflow_ui/lib/src/widgets/cards/podcast_card.dart` |

## Quick Rules for Me (Claude)

**Rule 1: Package-first thinking**
- Feature presentation в†’ `audiflow_app/lib/features/`
- Business logic, repositories, data sources, models в†’ `audiflow_domain/lib/src/features/`
- Podcast RSS parsing, feed models в†’ `audiflow_podcast/lib/src/`
- Shared UI components в†’ `audiflow_ui/lib/src/`
- Core utilities в†’ `audiflow_core/lib/src/`

**Rule 2: Layer separation across packages**
- Never import `audiflow_app` into any other package
- `audiflow_core` is the foundation - no dependencies on other packages
- `audiflow_podcast` depends only on `audiflow_core` - isolated RSS parsing
- `audiflow_domain` contains both business logic AND data access for performance

**Rule 3: File naming**
- Repositories: `{entity}_repository.dart` (interface) / `{entity}_repository_impl.dart` (implementation)
- Services: `{feature}_service.dart`
- Controllers: `{screen}_controller.dart`
- Screens: `{screen}_screen.dart`
- Widgets: `{component}_widget.dart` or `{component}.dart`
- Models: `{entity}.dart` (extend Table for Drift tables, use @freezed for non-persisted models)

**Rule 4: Model annotations**
- Persisted data в†’ extend `Table` class (Drift generates data classes)
- Transient/computed data в†’ use `@freezed`
- Never create separate DTOs - use Drift-generated data classes directly everywhere

## Testing Strategy
- **Unit tests**: All services, repositories, business logic in respective packages
- **Widget tests**: All screens, complex widgets in `audiflow_app` and `audiflow_ui`
- **Integration tests**: Critical user flows in `audiflow_app`
- Location: Each package's `test/` directory mirrors `lib/` structure

## Package Exports

Each package must have a main export file that exposes its public API:
- `audiflow_core/lib/audiflow_core.dart`
- `audiflow_domain/lib/audiflow_domain.dart`
- `audiflow_podcast/lib/audiflow_podcast.dart`
- `audiflow_ui/lib/audiflow_ui.dart`

Keep internal implementation in `lib/src/` and only export what's needed.

## Performance Benefits of Merged Architecture

**Why we merged data + domain:**
1. **Zero mapping overhead** - Drift models used directly, no DTO conversions
2. **Simpler codebase** - One less layer to navigate
3. **Better IDE support** - Direct navigation from UI to models
4. **Faster development** - Less boilerplate, fewer files
5. **Type safety maintained** - Repository pattern still enforces contracts

**Trade-offs accepted:**
- Less flexibility to swap Drift (but we're committed to it)
- Repository implementations in same package as interfaces (acceptable for this use case)
